# seed-data

현실적이고 **멱등한** 시드 데이터를 만든다. 스키마 제약을 위반하지 않고, 두 번 돌려도
중복·오염이 없어야 한다.

## 절차

### 1. 대상·범위 확정
무엇을 위한 데이터인지 정한다 (어떤 기능/테스트, 어떤 엔티티, 대략 몇 건).
규모 프로파일을 정하면 좋다: `small`(스모크) / `medium`(개발) / `large`(부하 감각).

### 2. 스키마·제약 파악
**추측하지 말고 실제 정의를 읽는다.**
- 스키마 출처: 마이그레이션 파일, ORM/엔티티 정의, DDL, JSON 스키마, 타입 정의 등
- 확인할 것: 필수/널 허용, 타입, 유니크·외래키 제약, enum·체크 제약, 기본값,
  엔티티 간 관계(부모→자식 생성 순서)
- 기존 시드/픽스처가 있으면 1~2개 읽어 형식·관례·삽입 방식을 따른다.

### 3. 멱등 전략 결정
재실행 안전성을 어떻게 보장할지 정한다:
- 식별 가능한 시드 키 사용 (예: 고정 ID 범위/접두사, `seed_*` 태그)
- upsert / `ON CONFLICT DO NOTHING` / "있으면 건너뛰기"
- 또는 "해당 시드 범위만 삭제 후 재삽입"
명령으로 데이터를 만들 거면 멱등성을 코드로 보장한다.

### 4. 생성
관계 순서대로(부모 먼저) 데이터를 만든다. 값은 그럴듯하게:
- 도메인 불변식 준수 (예: 금액 ≥ 0, 종료 ≥ 시작, 합계 일치)
- 시간값은 고정/상대 기준으로 (테스트 깨짐 방지 — 임의의 현재시각 남발 금지)
- 비밀번호 등은 해당 시스템 방식으로 처리, 평문 비밀 하드코딩 금지

### 5. 검증
- 제약 위반 없이 적재되는지 실제로 한 번 실행해 본다(가능하면).
- **두 번째 실행이 안전한지 확인한다** — 중복·에러 없이 멱등인지.
- 핵심 불변식이 지켜졌는지 확인(개수·합계 등).

### 6. 보고
무엇을 몇 건, 어떤 멱등 전략으로 만들었는지. 재실행/정리 방법 한 줄. 실제로 돌려보지
못했으면 그렇다고 명시한다.

## 원칙
- 운영 데이터에는 절대 적용하지 않는다 — 대상이 개발/테스트 환경인지 먼저 확인한다.
- 스키마 제약을 우회하려고 데이터를 왜곡하지 않는다. 막히면 스키마 쪽 문제를 보고한다.
- 기존 픽스처 스타일 > 내 선호.
